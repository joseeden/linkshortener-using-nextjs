# Security Audit Report — Link Shortener

**Date:** May 3, 2026
**Reviewer:** GitHub Copilot (automated security review)
**Repository:** linkshortener-using-nextjs

---

## Summary

This security audit covers the full source codebase of the Link Shortener application, a Next.js 16 (App Router) web application backed by PostgreSQL (Neon serverless), Drizzle ORM, and Clerk for authentication. The review focused on authentication flows, data handling, input validation, server actions, redirect logic, dependency security, and sensitive data exposure.

Files reviewed include all non-ignored TypeScript/TSX source files, configuration files (`next.config.ts`, `drizzle.config.ts`, `proxy.ts`), database layer (`db/`, `data/`), server actions (`app/dashboard/actions.ts`), route handlers (`app/api/redirect/[slug]/route.ts`), and `package.json` / lock files.

`.env` and `.env.local` files are correctly excluded by `.gitignore` and were not present in the repository; they must be reviewed manually in the deployment environment.

## Findings

| ID | Description | Location | Severity | Recommendation |
|----|-------------|----------|----------|----------------|
| 1 | **X-Forwarded-For IP spoofing bypasses rate limiting** — The redirect rate limiter extracts the client IP from the `x-forwarded-for` header, which is fully attacker-controlled. A malicious client can set an arbitrary value in this header on every request to obtain a fresh rate limit window, effectively bypassing the 30 req/min cap on the public redirect endpoint entirely. | [proxy.ts#L29](../proxy.ts#L29) | Medium | Validate or sanitize the `x-forwarded-for` header against a trusted set of reverse-proxy CIDRs, or rely on platform-provided trusted IP headers (e.g., Vercel's `x-real-ip` injected by the edge). Consider moving rate limiting to an edge-middleware-aware store (e.g., Upstash Redis) that cannot be influenced by client headers. |
| 2 | **In-memory rate limit state is non-persistent across serverless instances** — `rateLimitMap` is a module-level `Map`. In serverless/edge deployments (e.g., Vercel Edge Functions), each cold-start creates a new in-memory state. Multiple concurrent instances each maintain independent counters, meaning the effective global rate limit is `RATE_LIMIT × instance_count` rather than `RATE_LIMIT`. This degrades the DoS protection the rate limiter is intended to provide. | [proxy.ts#L7](../proxy.ts#L7) | Medium | Replace the in-memory store with a distributed, atomic counter (e.g., Upstash Redis with a sliding-window algorithm) to guarantee rate limiting holds across all instances. |
| 3 | **Missing HTTP security response headers** — `next.config.ts` does not configure security headers such as `Content-Security-Policy`, `X-Frame-Options`, `X-Content-Type-Options`, `Strict-Transport-Security`, or `Referrer-Policy`. Without these headers, browsers may permit clickjacking, MIME-type sniffing, and cross-origin data leakage. | [next.config.ts](../next.config.ts) | Medium | Add a `headers()` export in `next.config.ts` that sets at minimum: `X-Frame-Options: DENY`, `X-Content-Type-Options: nosniff`, `Referrer-Policy: strict-origin-when-cross-origin`, `Strict-Transport-Security: max-age=63072000; includeSubDomains`, and a scoped `Content-Security-Policy`. |
| 4 | **Vulnerable transitive dependency: PostCSS < 8.5.10 (GHSA-qx2v-qp2m-jg93, CVSSv3 6.1)** — Both `next` (main package) and `@clerk/ui` bundle an outdated PostCSS version that contains an XSS vulnerability: unescaped `</style>` tags in its CSS Stringify output can inject arbitrary script blocks into server-rendered HTML. | [package.json#L21](../package.json#L21) | Medium | Upgrade `@clerk/ui` to `≥ 1.7.0` (the fix is available as a semver-major update). For `next`, monitor upstream for a patched release that updates the bundled PostCSS. Run `npm audit fix --force` in a test environment after reviewing breaking changes. |
| 5 | **Vulnerable transitive dependency: esbuild ≤ 0.24.2 (GHSA-67mh-4wv8-2f99, CVSSv3 5.3)** — An outdated `esbuild` bundled inside `@esbuild-kit/core-utils` (pulled by `drizzle-kit`) allows any website to send requests to the local development server and read its responses, exposing local file contents and environment details to a malicious web page opened in the same browser as the running dev server. This affects only the development environment; production builds are not impacted. | [package.json#L38](../package.json#L38) | Low | Upgrade `drizzle-kit` to `≥ 0.18.1` (semver-major; review migration guide before upgrading). Alternatively, avoid opening untrusted web pages while running `npm run dev`, and restrict the dev server to `localhost` only. |
| 6 | **Non-null assertions on environment variables in `seed.ts` suppress startup errors** — `db/seed.ts` accesses `process.env.DATABASE_URL!` and `process.env.SEED_USER_ID!` using TypeScript non-null assertion operators without a runtime guard. Unlike `db/index.tsx` (which throws a descriptive error when `DATABASE_URL` is absent), `seed.ts` will silently pass `undefined` to the database driver, resulting in an opaque driver-level error. | [db/seed.ts#L6](../db/seed.ts#L6) | Low | Add explicit runtime checks before using these values, mirroring the pattern in `db/index.tsx`: `if (!process.env.DATABASE_URL) throw new Error('DATABASE_URL is not set');` |
| 7 | **Default application metadata exposes scaffolding fingerprint** — `app/layout.tsx` retains the `title: "Create Next App"` and `description: "Generated by create next app"` metadata from the project scaffold. While not directly exploitable, these strings confirm the framework and version family to an attacker performing reconnaissance. | [app/layout.tsx#L22](../app/layout.tsx#L22) | Low | Update the `metadata` object with application-specific title and description values. |

## Items Reviewed and Found Secure

The following areas were inspected and determined to be acceptably secure given the framework protections and code patterns in use:

- **SQL Injection** — All database queries are performed via Drizzle ORM's parameterised query builder (`eq()`, `and()`, `sql` tagged template). No raw SQL string interpolation was found in application code.
- **Authentication & Authorization** — Clerk's `auth()` helper is called server-side at the start of every server action and in the dashboard page. All write operations (`createLink`, `updateLink`, `deleteLink`) include the authenticated `userId` in the SQL `WHERE` clause, preventing Insecure Direct Object Reference (IDOR).
- **Open Redirect Protocol Validation** — `app/api/redirect/[slug]/route.ts` validates that the stored URL's protocol is `http:` or `https:` before issuing the redirect, blocking `javascript:` or `data:` URI exploitation.
- **Input Validation on Server Actions** — All server actions validate input with Zod schemas before processing. URL, slug, and ID fields are strictly typed and sanitised at the boundary.
- **Slug Injection** — The redirect route handler validates the slug against a strict allowlist regex (`/^[a-zA-Z0-9_-]+$/`) before performing any database lookup.
- **Hardcoded Secrets** — No secrets, credentials, or API keys were found hardcoded in any non-ignored source file. Sensitive values are referenced only via `process.env.*` and the `.env` file is correctly excluded from version control by `.gitignore`.
- **Clickjacking on External Links** — Dashboard external links use `rel="noopener noreferrer"` and `target="_blank"`, preventing tab-napping and referrer leakage.
- **CSRF** — Next.js 13+ Server Actions are CSRF-protected by the framework via `SameSite` cookie policy and origin validation. No additional CSRF tokens are required.

---

## Conclusion

The application's core security posture is solid: authentication is delegated entirely to Clerk, database access uses parameterised queries through Drizzle ORM, server actions enforce authentication and ownership checks, and redirect targets are protocol-validated. No hardcoded secrets were found in tracked files.

The most actionable findings are the rate-limiter reliability issues (IDs 1 and 2), which together mean the rate limiting on the public redirect endpoint provides limited real-world protection in a distributed deployment. Adding `Content-Security-Policy` and other security headers (ID 3) and upgrading the PostCSS dependency (ID 4) round out the Medium-severity remediation backlog.

The Low-severity findings (IDs 5–7) are primarily hygiene issues with minimal direct exploitation potential.

**Overall risk level: Medium** — no critical or high-severity issues were identified.
