# Security Audit Report

**Report Title:** Link Shortener — Security Audit  
**Date Performed:** May 3, 2026  
**Auditor:** GitHub Copilot (automated review)

---

## Summary

This security audit covers the full codebase of the `linkshortener` Next.js 16 application. The scope includes all non-ignored source files: server actions, route handlers, authentication middleware, database layer, UI components, configuration files, and `package.json` (including lock file dependency analysis via `npm audit`).

Files excluded from analysis (ignored by `.gitignore`): `.env`, `.env.local`, `.env.*.local`.

Key areas examined:
- Authentication and route protection (Clerk-based)
- Data input validation and sanitisation
- Database access patterns (Drizzle ORM + Neon serverless)
- Open-redirect and redirect handler logic
- HTTP response security headers
- Sensitive data handling and environment variable usage
- Third-party dependency vulnerabilities (`npm audit`)

## Findings

| ID | Description | Location | Severity | Recommendation |
|----|-------------|----------|----------|----------------|
| 1 | **Missing HTTP Security Headers** — `next.config.ts` does not configure any HTTP security headers (`Content-Security-Policy`, `X-Frame-Options`, `X-Content-Type-Options`, `Referrer-Policy`, `Strict-Transport-Security`). Without `X-Frame-Options` or a CSP `frame-ancestors` directive the application can be embedded in a third-party iframe, enabling clickjacking against authenticated users. Without a CSP, any future XSS vulnerability has no browser-level mitigation. | [next.config.ts](../next.config.ts) | Medium | Add a `headers()` export to `next.config.ts` that sets at minimum: `X-Frame-Options: DENY`, `X-Content-Type-Options: nosniff`, `Referrer-Policy: strict-origin-when-cross-origin`, and a `Content-Security-Policy` appropriate for the app's asset origins. |
| 2 | **No Rate Limiting on Public Redirect Endpoint** — The `GET /api/redirect/[slug]` handler is public and unauthenticated. There is no rate limiting or request throttling. An attacker can systematically enumerate all valid slugs (information gathering) or make bulk requests to artificially inflate click counters, with no cost. | [app/api/redirect/\[slug\]/route.ts](../app/api/redirect/%5Bslug%5D/route.ts) | Medium | Implement rate limiting in `proxy.ts` (Clerk middleware layer) using an in-memory or edge-compatible store (e.g., Vercel's `@vercel/kv` or an upstash Redis rate limiter) to limit unauthenticated requests per IP per minute on the `/:slug` and `/api/redirect/*` paths. |
| 3 | **Vulnerable `next` Dependency (PostCSS XSS — GHSA-qx2v-qp2m-jg93)** — `next@16.2.4` bundles `postcss < 8.5.10` which has a confirmed XSS vulnerability (CWE-79, CVSS 6.1) via unescaped `</style>` in its CSS Stringify output. `npm audit` reports this as moderate with no fix available in the current range. The vulnerability is not directly exploitable in this application because no user-supplied CSS is processed through PostCSS at runtime; however the bundled vulnerable package is present on disk and the risk increases if usage patterns change. | [package.json](../package.json) | Medium | Monitor Next.js releases and upgrade to `next >= 16.3.0` once available. Track advisory GHSA-qx2v-qp2m-jg93. |
| 4 | **`@clerk/ui` Pulls in Vulnerable Solana Wallet Adapter Chain (uuid CWE-787 — GHSA-w5hq-g745-h8pq)** — `@clerk/ui >= 0.3.25-canary.v20250218163602` introduces a chain of Solana wallet adapter packages (`@solana/web3.js`, `rpc-websockets`, `uuid < 14.0.0`). The `uuid` package in this chain has a missing buffer-bounds check (CWE-787, CWE-1285) when the optional `buf` argument is used. The chain is transitive and unlikely to be triggered in normal Clerk authentication flows, but the dependency is unnecessary for a link-shortener application. `npm audit` reports a fix: downgrade `@clerk/ui` to `0.3.24`. | [package.json](../package.json) | Medium | Pin `@clerk/ui` to `0.3.24` in `package.json` (breaking major change per npm audit). Run `npm install` and verify Clerk authentication still functions correctly. |
| 5 | **Raw Database Error Messages Returned to Client** — In `createLinkAction` (line 51) and `updateLinkAction` (line 86), if the database throws an error that is not a slug-uniqueness error, the raw `err.message` string is returned directly to the calling client. Database driver error messages can expose internal schema details, connection strings, or table names. | [app/dashboard/actions.ts](../app/dashboard/actions.ts#L51) | Low | Return a generic message (e.g., `'An unexpected error occurred. Please try again.'`) for unclassified database errors. Log the full error server-side with a correlation ID if debugging is needed. |
| 6 | **No Runtime Validation for Critical Environment Variables** — `db/index.tsx` (line 3) and `drizzle.config.ts` (line 8) use the TypeScript non-null assertion (`!`) on `process.env.DATABASE_URL`. If the variable is absent at runtime the driver receives `undefined`, potentially causing cryptic connection failures rather than a clear startup error. | [db/index.tsx](../db/index.tsx#L3) | Low | Add an explicit runtime guard at module load: `if (!process.env.DATABASE_URL) throw new Error('DATABASE_URL is not set')`. This provides a fast-fail with a clear error message instead of a silent downstream failure. |
| 7 | **Seed Script Lacks Production Safeguard** — `db/seed.ts` inserts test records using whatever `DATABASE_URL` and `SEED_USER_ID` are set in the current environment. If accidentally executed against a production database (e.g., during a CI/CD pipeline misconfiguration), it would insert test link records under an arbitrary user ID. | [db/seed.ts](../db/seed.ts) | Low | Add an explicit guard at the top of the script: check that `NODE_ENV !== 'production'` or that a specific `ALLOW_SEED=true` flag is set, and abort with a warning otherwise. |

## Non-Findings (Mitigated by Framework or Code Structure)

The following potential concerns were evaluated and found to be **not exploitable** in this codebase:

- **SQL Injection** — All database queries use Drizzle ORM's parameterized query builder (`eq`, `and`, `desc`, `sql` tagged template). No raw string concatenation into SQL is present.
- **Open Redirect (Protocol Bypass)** — The redirect handler validates that the protocol of stored URLs is `http:` or `https:` (line 44, `route.ts`), and the creation/update actions enforce the same constraint via Zod schema before persisting to the database. `javascript:` and `data:` URI schemes are blocked.
- **CSRF on Server Actions** — Next.js 16 App Router Server Actions include built-in CSRF protection via Origin header validation. No additional CSRF token implementation is needed.
- **Authentication Bypass** — Route protection is enforced in `proxy.ts` (Clerk middleware) at the edge before any page or API handler runs. The dashboard server component also calls `auth()` and returns `null` for unauthenticated requests as a secondary guard.
- **Insecure Direct Object Reference (IDOR)** — All mutating actions (`updateLink`, `deleteLink`) pass `userId` from the server-side `auth()` call as a WHERE condition alongside the record `id`, ensuring a user cannot modify another user's links.
- **XSS in `notFoundHtml`** — The `message` parameter interpolated into HTML is only ever called with two hardcoded string literals (`'Invalid link'`, `'Link not found'`); no user input reaches this function.
- **Hardcoded Secrets** — No API keys, credentials, or secrets are present in any tracked file. All sensitive values are loaded from environment variables; `.env` files are properly listed in `.gitignore`.
- **Insecure Redirect After Login** — Clerk's `signInForceRedirectUrl` and `signUpForceRedirectUrl` are set to `/dashboard`, preventing open-redirect abuse via crafted callback URLs.

---

## Conclusion

The codebase demonstrates a solid security baseline: authentication is fully delegated to Clerk with proper route protection, all database access is parameterised via Drizzle ORM, URL inputs are validated with Zod at both the action and redirect layers, and no secrets are committed to source control.

The most actionable improvements are adding HTTP security response headers (ID 1) and rate limiting on the public redirect endpoint (ID 2), both of which are straightforward additions to `next.config.ts` and `proxy.ts` respectively. The dependency vulnerabilities (IDs 3 and 4) should be tracked and resolved as upstream fixes become available, with `@clerk/ui@0.3.24` being an immediately available fix for ID 4.

Overall, the codebase presents a **low-to-medium risk posture**. No critical or high-severity vulnerabilities were identified.
