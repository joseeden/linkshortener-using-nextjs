## Finding Details

### ID 1 — Missing HTTP Security Headers

`next.config.ts` defines only URL rewrites; no `headers()` function is present. Without these headers, the application is exposed to:

- **Clickjacking** — no `X-Frame-Options` or `frame-ancestors` CSP directive
- **MIME sniffing** — no `X-Content-Type-Options: nosniff`
- **Downgrade attacks** — no `Strict-Transport-Security`
- **Referrer leakage** — no `Referrer-Policy`
- **XSS / injection vectors** — no `Content-Security-Policy`

**Recommended addition to `next.config.ts`:**

```ts
async headers() {
  return [
    {
      source: '/(.*)',
      headers: [
        { key: 'X-Frame-Options', value: 'DENY' },
        { key: 'X-Content-Type-Options', value: 'nosniff' },
        { key: 'Referrer-Policy', value: 'strict-origin-when-cross-origin' },
        {
          key: 'Strict-Transport-Security',
          value: 'max-age=63072000; includeSubDomains; preload',
        },
        {
          key: 'Content-Security-Policy',
          value: [
            "default-src 'self'",
            "script-src 'self' 'unsafe-inline' https://clerk.com https://*.clerk.accounts.dev",
            "style-src 'self' 'unsafe-inline'",
            "img-src 'self' data: https:",
            "font-src 'self'",
            "connect-src 'self' https://clerk.com https://*.clerk.accounts.dev",
            "frame-ancestors 'none'",
          ].join('; '),
        },
      ],
    },
  ];
},
```

> Adjust the CSP `script-src` and `connect-src` directives to match your Clerk frontend API domain. Verify with browser DevTools before enabling in production.

### ID 2 — In-Memory Rate Limiter

The `rateLimitMap` in `proxy.ts` is a module-level `Map`. In serverless runtimes (Vercel, AWS Lambda), each function instance has its own memory. The 30 req/min limit therefore applies per instance, not globally. A warning comment is already present in the code. The actionable fix is:

```
npm install @upstash/ratelimit @upstash/redis
```

Then replace `checkRateLimit` with an Upstash sliding-window limiter backed by a Redis database.

---

### ID 3 — `x-forwarded-for` Fallback

Currently `proxy.ts` prefers `x-real-ip` and falls back to `x-forwarded-for`. On Vercel, `x-real-ip` is always injected and `x-forwarded-for` is overwritten by the edge, so the fallback is safe in that environment. On any other deployment target the fallback is attacker-controlled. A warning comment is present in the code; no additional code change is possible without infrastructure support.

---

### ID 4 — 301 Permanent Redirect Caching

The schema defaults `isPermanent` to `false`, so this risk applies only when a link is explicitly created with `isPermanent: true`. The concern is that browsers (and proxies) cache `301` responses indefinitely. If the link destination is later updated or the link deleted, cached clients will still redirect to the original (potentially stale or malicious) URL.

---

### ID 5 — Server Action Rate Limiting

All three write server actions (`createLinkAction`, `updateLinkAction`, `deleteLinkAction`) require a valid Clerk `userId`, providing a strong authentication barrier. The residual risk is an authenticated user (or a compromised account) making high-frequency requests to create slugs, exhaust quota, or probe collision behaviour. The rate-limiting infrastructure introduced for ID 2 (Upstash Redis) would naturally extend to cover this case.

## Mitigated / Not Reported

The following areas were reviewed and found to be adequately protected:

| Area | Status |

| Dependency vulnerabilities | `npm audit` — 0 vulnerabilities (1,087 packages). `postcss`, `esbuild`, `uuid` overrides in place. |

| Open redirect | Users must be authenticated to create links; URLs are validated (http/https only) at creation time. |

## Conclusion

The codebase maintains a solid security baseline: zero dependency CVEs, all database queries are parameterised, every write operation is authenticated and validated, and previously identified issues from prior audits have been remediated or documented.

**Three medium-severity issues remain open** — missing HTTP security headers (`next.config.ts`), non-distributed in-memory rate limiting (`proxy.ts`), and the `x-forwarded-for` spoofing fallback — all of which are actionable. The two low-severity findings (permanent redirect caching and lack of server-action rate limiting) are low-probability risks given the current deployment model and can be addressed when the Upstash Redis infrastructure is introduced.

**Priority recommendation:** Address ID 1 (HTTP security headers) first — it is a low-effort, high-value code-only change requiring no external infrastructure.
