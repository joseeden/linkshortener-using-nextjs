# Security Audit Report — Link Shortener

**Date:** May 3, 2026  
**Reviewer:** GitHub Copilot (automated security review)  
**Repository:** linkshortener-using-nextjs

---

## Summary

This security audit covers the full source codebase of the Link Shortener application, a Next.js 16 (App Router) web application backed by PostgreSQL (Neon serverless), Drizzle ORM, and Clerk for authentication. The review reflects the current state of the codebase, including remediations applied following the previous audit (`security-report-20260503-172426.md`).

Files reviewed include all non-ignored TypeScript/TSX source files, configuration files (`next.config.ts`, `drizzle.config.ts`, `proxy.ts`), database layer (`db/`, `data/`), server actions (`app/dashboard/actions.ts`), route handler (`app/api/redirect/[slug]/route.ts`), and `package.json`.

`.env` and `.env.local` files are correctly excluded by `.gitignore` and were not present in the repository; they must be reviewed manually in the deployment environment.

## Findings

| ID | Description | Location | Severity | Recommendation |
|----|-------------|----------|----------|----------------|
| 1 | **Missing HTTP security response headers** — `next.config.ts` does not configure security headers such as `Content-Security-Policy`, `X-Frame-Options`, `X-Content-Type-Options`, `Strict-Transport-Security`, or `Referrer-Policy`. Without these headers, browsers may permit clickjacking, MIME-type sniffing, and cross-origin data leakage. | [next.config.ts](../next.config.ts) | Medium | Add a `headers()` export in `next.config.ts` that sets at minimum: `X-Frame-Options: DENY`, `X-Content-Type-Options: nosniff`, `Referrer-Policy: strict-origin-when-cross-origin`, `Strict-Transport-Security: max-age=63072000; includeSubDomains`, and a scoped `Content-Security-Policy`. |
| 2 | **In-memory rate limit state is non-persistent across serverless instances** — `rateLimitMap` is a module-level `Map`. In serverless/edge deployments (e.g., Vercel), each cold-start creates a new in-memory state. Multiple concurrent instances each maintain independent counters, meaning the effective global rate limit is `RATE_LIMIT × instance_count` rather than `RATE_LIMIT`. A warning comment was added in the previous remediation cycle but the underlying issue remains unresolved. | [proxy.ts#L11](../proxy.ts#L11) | Medium | Replace the in-memory store with a distributed, atomic counter (e.g., Upstash Redis with a sliding-window algorithm via `@upstash/ratelimit`) to guarantee rate limiting holds across all instances. Requires `UPSTASH_REDIS_REST_URL` and `UPSTASH_REDIS_REST_TOKEN` environment variables. |
| 3 | **Vulnerable transitive dependency: PostCSS (GHSA-qx2v-qp2m-jg93, CVSSv3 6.1)** — `next` bundles an outdated PostCSS version that contains an XSS vulnerability: unescaped `</style>` tags in its CSS Stringify output can inject arbitrary script blocks into server-rendered HTML. Confirmed present via `npm audit`. | [package.json](../package.json) | Medium | Monitor the upstream `next` release cycle for a patched version that updates its bundled PostCSS. As a workaround, add `"overrides": { "postcss": ">=8.5.10" }` to `package.json` to force the patched version where possible, and test thoroughly. |
| 4 | **`x-forwarded-for` fallback retains IP spoofing risk in non-platform environments** — The rate limiter now correctly prefers `x-real-ip` (a Vercel platform-injected header). However, `x-forwarded-for` remains as a fallback. In non-Vercel deployments (self-hosted, local dev with a custom proxy) where `x-real-ip` is not injected, a client can still set an arbitrary `x-forwarded-for` value on every request, obtaining a fresh rate-limit window and bypassing the cap entirely. | [proxy.ts#L34](../proxy.ts#L34) | Medium | When deploying behind a trusted reverse proxy other than Vercel, configure the proxy to inject a tamper-proof IP header (e.g., `x-real-ip`) and remove any existing `x-forwarded-for` value from untrusted clients before it reaches the application. Alternatively, the long-term fix (ID 2 — Upstash Redis) eliminates this attack surface entirely. |
| 5 | **Vulnerable transitive dependency: esbuild ≤ 0.24.2 (GHSA-67mh-4wv8-2f99, CVSSv3 5.3)** — `drizzle-kit` transitively pulls `@esbuild-kit/core-utils@3.3.2` → `esbuild@0.18.20`. This allows any website to send requests to the local development server and read its responses, exposing local file contents and environment details. This affects only the development environment; production builds are not impacted. Confirmed present via `npm audit`; no non-breaking automated fix is currently available. | [package.json](../package.json) | Low | Avoid opening untrusted web pages while running `npm run dev`. Monitor `drizzle-kit` for a release that upgrades its bundled esbuild past `0.24.2`. Consider restricting the dev server to `localhost` only. |
| 6 | **Non-null assertion on `DATABASE_URL` in `drizzle.config.ts` suppresses startup errors** — `drizzle.config.ts` accesses `process.env.DATABASE_URL!` using a TypeScript non-null assertion without a runtime guard. If the variable is absent, the database driver receives `undefined` and emits an opaque driver-level error rather than a descriptive startup failure. This contrasts with the now-corrected `db/index.tsx` and `db/seed.ts`, which both include explicit runtime checks. | [drizzle.config.ts#L9](../drizzle.config.ts#L9) | Low | Add an explicit runtime check before the `defineConfig` call, mirroring the pattern already used in `db/index.tsx`: `if (!process.env.DATABASE_URL) throw new Error('DATABASE_URL is not set');` |

## Items Reviewed and Found Secure

The following areas were inspected and determined to be acceptably secure given the framework protections and code patterns in use:

- **SQL Injection** — All database queries are performed via Drizzle ORM's parameterised query builder (`eq()`, `and()`, `sql` tagged template). No raw SQL string interpolation was found in application code.
- **Authentication & Authorization** — Clerk's `auth()` helper is called server-side at the start of every server action and in the dashboard page. All write operations (`createLink`, `updateLink`, `deleteLink`) include the authenticated `userId` in the SQL `WHERE` clause, preventing Insecure Direct Object Reference (IDOR).
- **Open Redirect Protocol Validation** — `app/api/redirect/[slug]/route.ts` validates that the stored URL's protocol is `http:` or `https:` before issuing the redirect, blocking `javascript:` or `data:` URI exploitation.
- **Input Validation on Server Actions** — All server actions validate input with Zod schemas before processing. URL, slug, and ID fields are strictly typed and sanitised at the boundary.
- **Slug Injection** — The redirect route handler validates the slug against a strict allowlist regex (`/^[a-zA-Z0-9_-]+$/`) before performing any database lookup.
- **Hardcoded Secrets** — No secrets, credentials, or API keys were found hardcoded in any non-ignored source file. Sensitive values are referenced only via `process.env.*` and the `.env` file is correctly excluded from version control by `.gitignore`.
- **Clickjacking on External Links** — Dashboard external links use `rel="noopener noreferrer"` and `target="_blank"`, preventing tab-napping and referrer leakage.
- **CSRF** — Next.js 13+ Server Actions are CSRF-protected by the framework via `SameSite` cookie policy and origin validation. No additional CSRF tokens are required.
- **`notFoundHtml` Template Function** — The inline HTML template in `route.ts` uses string interpolation for `message`, but all call sites pass static string literals (`'Invalid link'`, `'Link not found'`), so no user-controlled input can reach the template. Not exploitable with current usage.
- **IP Header Preference (ID 1 from previous report)** — The rate limiter now correctly prefers the platform-injected `x-real-ip` over the client-controllable `x-forwarded-for`. The primary spoofing vector on Vercel is fully mitigated; a residual risk in non-platform environments is captured as finding ID 4 above.
- **`db/seed.ts` Environment Variable Guards** — `seed.ts` now includes explicit runtime checks for `DATABASE_URL` and `SEED_USER_ID` before use, matching the pattern in `db/index.tsx`. Previously flagged; now resolved.
- **Application Metadata** — `app/layout.tsx` metadata has been updated from scaffold defaults to application-specific values. Previously flagged; now resolved.

## Conclusion

Since the previous audit, three issues have been successfully remediated (IP header preference, seed.ts runtime guards, scaffold metadata). The application's core security posture remains solid: authentication is fully delegated to Clerk, database access uses parameterised queries through Drizzle ORM, server actions enforce authentication and ownership checks, and redirect targets are protocol-validated.

The highest-priority outstanding items are the missing HTTP security headers (ID 1), which provide broad browser-level protections at low implementation cost, and the distributed rate limiting gap (ID 2), which requires external infrastructure (Upstash Redis) to fully resolve. The `x-forwarded-for` fallback (ID 4) is mitigated on Vercel but warrants attention in any other deployment context.

The two Low-severity findings (IDs 5–6) are hygiene issues with limited direct exploitation potential in their current context.

**Overall risk level: Medium** — no critical or high-severity issues were identified.
