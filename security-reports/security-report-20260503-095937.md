# Security Audit Report — Link Shortener

**Date:** May 3, 2026  
**Reviewer:** GitHub Copilot (automated security review)  
**Repository:** linkshortener-using-nextjs

---

## Summary

This security audit covers the full source codebase of the Link Shortener application following multiple remediation cycles. The application is a Next.js 16 (App Router) web application backed by PostgreSQL (Neon serverless), Drizzle ORM, and Clerk for authentication.

Files reviewed include all non-ignored TypeScript/TSX source files, configuration files (`next.config.ts`, `drizzle.config.ts`, `proxy.ts`), database layer (`db/`, `data/`), server actions (`app/dashboard/actions.ts`), route handler (`app/api/redirect/[slug]/route.ts`), and `package.json`.

Notable changes since the previous audit (`security-report-20260503-172426.md`): `@clerk/ui` was upgraded to `1.7.0`; `"overrides"` were added to `package.json` forcing `postcss >= 8.5.10` and `esbuild >= 0.25.0`; `db/seed.ts` gained explicit env-var runtime guards; `app/layout.tsx` metadata was updated; and `proxy.ts` now prefers `x-real-ip` over `x-forwarded-for`.

`.env` and `.env.local` files are correctly excluded by `.gitignore` and were not present in the repository; they must be reviewed manually in the deployment environment.

## Findings

| ID | Description | Location | Severity | Recommendation |
|----|-------------|----------|----------|----------------|
| 1 | **In-memory rate limit state is non-persistent across serverless instances** — `rateLimitMap` is a module-level `Map`. In serverless/edge deployments (e.g., Vercel), each cold-start creates a new in-memory state. Multiple concurrent instances each maintain independent counters, meaning the effective global rate limit is `RATE_LIMIT × instance_count` rather than `RATE_LIMIT`. A `WARNING` comment was added in the previous cycle but the code-level risk is unresolved. | [proxy.ts#L11](../proxy.ts#L11) | Medium | Replace the in-memory store with a distributed, atomic counter using `@upstash/ratelimit` and `@upstash/redis` with a sliding-window algorithm. Requires `UPSTASH_REDIS_REST_URL` and `UPSTASH_REDIS_REST_TOKEN` environment variables. |
| 2 | **Missing HTTP security response headers** — `next.config.ts` does not configure `Content-Security-Policy`, `X-Frame-Options`, `X-Content-Type-Options`, `Strict-Transport-Security`, or `Referrer-Policy`. Without these, browsers may permit clickjacking, MIME-type sniffing, and cross-origin data leakage. | [next.config.ts](../next.config.ts) | Medium | Add a `headers()` export in `next.config.ts` setting at minimum: `X-Frame-Options: DENY`, `X-Content-Type-Options: nosniff`, `Referrer-Policy: strict-origin-when-cross-origin`, `Strict-Transport-Security: max-age=63072000; includeSubDomains`, and a scoped `Content-Security-Policy`. |
| 3 | **Vulnerable transitive dependency: uuid < 14.0.0 (GHSA-w5hq-g745-h8pq, CVSSv3 moderate)** — `@clerk/ui@1.7.0` pulls `@solana/wallet-adapter-base` → `@solana/web3.js` → `jayson` and `rpc-websockets`, both of which depend on `uuid < 14.0.0`. This version contains a missing buffer bounds check in `v3`/`v5`/`v6` UUID generation when a `buf` argument is provided, which can cause out-of-bounds memory writes. Confirmed present via `npm audit`. | [package.json](../package.json) | Medium | Add `"uuid": ">=14.0.0"` to the `"overrides"` section of `package.json` to force all transitive consumers to the patched version. Test thoroughly as this is a semver-major bump for uuid consumers. Alternatively, monitor `@clerk/ui` for a release that updates the Solana dependency chain. |
| 4 | **`x-forwarded-for` fallback retains IP spoofing risk in non-platform environments** — The rate limiter correctly prefers `x-real-ip` (Vercel platform-injected, not client-controllable). However, `x-forwarded-for` is still used as a fallback. In self-hosted or local-dev deployments without `x-real-ip` injection, a client can set an arbitrary `x-forwarded-for` value to bypass the rate limit on every request. | [proxy.ts#L33](../proxy.ts#L33) | Medium | In non-Vercel deployments, configure the reverse proxy to inject a tamper-proof IP header and strip any client-supplied `x-forwarded-for` before it reaches the application. The long-term fix (ID 1 — Upstash Redis) eliminates this entire attack surface. |
| 5 | **Non-null assertion on `DATABASE_URL` in `drizzle.config.ts` suppresses startup errors** — `drizzle.config.ts` uses `process.env.DATABASE_URL!` without a runtime guard. If the variable is absent, the database driver receives `undefined` and emits an opaque error. This contrasts with `db/index.tsx` and `db/seed.ts`, which both have explicit checks. | [drizzle.config.ts#L9](../drizzle.config.ts#L9) | Low | Add an explicit runtime check before `defineConfig`: `if (!process.env.DATABASE_URL) throw new Error('DATABASE_URL is not set');` — mirroring the pattern in `db/index.tsx`. |

## Items Reviewed and Found Secure

- **SQL Injection** — All database queries use Drizzle ORM's parameterised query builder (`eq()`, `and()`, `sql` tagged template). No raw SQL string interpolation found.
- **Authentication & Authorization** — Clerk's `auth()` is called server-side at the start of every server action and the dashboard page. All write operations include the authenticated `userId` in `WHERE` clauses, preventing IDOR.
- **Open Redirect Protocol Validation** — `app/api/redirect/[slug]/route.ts` validates `http:`/`https:` protocol before redirecting, blocking `javascript:` and `data:` URI exploitation.
- **Input Validation on Server Actions** — All server actions validate with Zod schemas. URL, slug, and ID fields are strictly typed and sanitised at the boundary.
- **Slug Injection** — The redirect route handler validates slugs against `/^[a-zA-Z0-9_-]+$/` before any database lookup.
- **Hardcoded Secrets** — No secrets, credentials, or API keys found in any non-ignored source file.
- **Clickjacking on External Links** — Dashboard links use `rel="noopener noreferrer"` and `target="_blank"`.
- **CSRF** — Next.js 13+ Server Actions are CSRF-protected by the framework via `SameSite` cookie policy and origin validation.
- **IP Header Preference** — `x-real-ip` is now preferred over `x-forwarded-for` in the rate limiter, mitigating the primary spoofing vector on Vercel. *(Previously finding ID 1; now resolved.)*
- **PostCSS XSS (GHSA-qx2v-qp2m-jg93)** — Resolved: `@clerk/ui` upgraded to `1.7.0` and `"overrides": { "postcss": ">=8.5.10" }` added. `npm audit` confirms no PostCSS vulnerabilities. *(Previously finding ID 4; now resolved.)*
- **esbuild dev vulnerability (GHSA-67mh-4wv8-2f99)** — Resolved: `"overrides": { "esbuild": ">=0.25.0" }` added. `npm audit` confirms no esbuild vulnerabilities. *(Previously finding ID 5; now resolved.)*
- **seed.ts non-null assertions** — Resolved: explicit `if (!process.env.X) throw new Error(...)` guards added in `db/seed.ts`. *(Previously finding ID 6; now resolved.)*
- **Scaffold metadata fingerprint** — Resolved: `app/layout.tsx` metadata updated to `"LinkShortener"` / `"Create and manage short links"`. *(Previously finding ID 7; now resolved.)*

---

## Conclusion

The remediation history shows steady improvement. Six of the seven findings from the previous audit have been resolved. The remaining open items are: the distributed rate-limiting gap (ID 1), which requires Upstash Redis infrastructure; missing HTTP security headers (ID 2), which is low-effort and high-impact; a newly surfaced `uuid` transitive vulnerability introduced by the `@clerk/ui` upgrade (ID 3), addressable via an `overrides` entry; the `x-forwarded-for` fallback residual risk in non-Vercel deployments (ID 4); and a minor `drizzle.config.ts` missing guard (ID 5).

No hardcoded secrets, SQL injection vectors, authentication bypasses, or open redirect exploits were found. The application's core security posture is solid.

**Overall risk level: Medium** — no critical or high-severity issues were identified.
